# 真实性边界与证据分级

> 核心规则：**Polish expression, not reality.** 润色表达，不润色事实。

---

## 一、真实性边界分类

对简历中每个重要声明，按以下 5 级分类：

| 状态 | 含义 | 处理方式 |
|------|------|---------|
| **可以写** | 现有证据充分支持 | 直接润色表达 |
| **谨慎写** | 有部分证据，但措辞需小心 | 降级措辞，限定范围 |
| **补证据后写** | 合理但缺乏证据 | 先补证据，再写进简历 |
| **不能写** | 无证据支持或虚假 | 删除或降级 |
| **无法判断** | 信息不足，需用户确认 | 向用户提问 |

### 常见降级对照表

| 不安全写法 | 安全写法 | 需要的证据 |
|-----------|---------|-----------|
| 主导系统建设 | 参与模块开发 / 负责某一子模块 | ownership 文档、代码、设计决策 |
| 上线生产环境 | 完成 demo / 内部试用 / 本地验证 | release 记录、流量、用户、日志 |
| 提升准确率 30% | 基于样例对比观察到效果改善 | 指标定义、baseline、报告 |
| 训练大模型 | 复现小模型训练 / 做 LoRA 微调 / 调用模型 API | dataset、config、logs、checkpoint |
| Agent 自动完成任务 | 在限定流程内完成工具调用和人工复核 | tool schema、logs、failure cases |
| 优化搜索排序 | 梳理 query-doc bad case / 复现 rerank baseline | query-doc labels、NDCG/MRR、baseline |
| 企业级应用 | 企业文档问答 demo / 内部原型 | production requirements、scale、reliability |
| 负责项目 | 参与项目某模块开发 | 代码 diff、commit 记录、任务分配 |
| 显著提升 | 对比分析 / 建立评估口径 | metric definition、baseline comparison |

### 危险词降级规则

以下词汇在没有对应证据时**必须降级**：

| 危险词 | 降级为 |
|--------|--------|
| 主导 | 参与 / 负责某模块 |
| 上线 | demo / 内部试用 / 本地验证 |
| 提升 X% | 对比分析 / 建立评估口径 |
| 训练大模型 | 复现小模型训练 / 学习训练流程 |
| 企业级 | demo / 内部原型 |
| 显著 | 可观察到 / 初步 |
| 负责 | 参与 / 协助 |
| 独立完成 | 在指导下完成 / 完成某一部分 |

### Non-Negotiables（不可协商的底线）

- 不写"主导"当证据只支持"参与"
- 不写"上线"当证据只支持 demo、本地运行或内部试用
- 不把开源学习写成工作经验，除非用户确实复现、修改并文档化了
- 不虚构实习、生产状态、指标、用户量、模型训练、排序提升或 ownership
- 如果材料不足，提问或产出保守报告，而非编造润色稿

## 二、证据分级（C0-C3）

| 级别 | 表达方式 | 所需证据 |
|------|---------|---------|
| **C0** | 了解 / 参与 | 笔记、解释、课程材料、项目文档 |
| **C1** | 负责模块 | 代码、README、截图、日志、任务笔记 |
| **C2** | 设计 / 优化 | 实验、对比、bad case、设计文档 |
| **C3** | 结果影响 | 指标定义、baseline、日志、报告、上线记录 |

**C3 级声明在没有指标定义和记录的情况下不允许写入简历。**

### 常见声明的最低证据要求

| 声明 | 最低证据 | 缺证据时的安全写法 |
|------|---------|-------------------|
| 优化 RAG 效果 | eval set、bad cases、检索/引用指标 | 整理 RAG bad case 并提出优化方向 |
| 提升搜索相关性 | query-doc labels、NDCG/MRR/CTR、baseline | 分析长尾 query 和相关性失败样例 |
| 支持线上业务 | release/内部试用/用户日志 | 完成内部 demo 或流程验证 |
| Agent 工具调用 | tool schema、logs、failure cases | 梳理工具调用流程与失败类型 |
| 训练/微调模型 | dataset、script、config、metrics | 复现训练流程或完成小规模微调实验 |

## 三、材料审计

当用户提供 materials 文件夹时，按以下规则审计：

### 提取内容
- 项目、实习、代码证据、指标、bad case、日志/截图、论文、竞赛/获奖、课程、开源贡献、ownership 边界

### 证据强度分级

| 级别 | 含义 | 示例 |
|------|------|------|
| **强** | 可直接支持简历声明 | code diff、报告、指标表、日志、README、demo 截图 |
| **中** | 有用但需解释 | 笔记、课程项目、部分实验、小规模对比 |
| **弱** | 不能单独支持强措辞 | 模糊记忆、最终截图、复制的 repo、无运行记录 |
| **缺失** | 需要但不存在 | 无指标、无 ownership、无可运行证明 |

### 审计规则
- 区分个人工作与团队/平台/开源工作
- 无支持的指标标记为缺失证据
- 生产声明除非有日志、release、用户或内部试用证据，否则标记为高风险
- 代码文件夹检查：README、tests、scripts、examples、commits、notebooks、logs
- PDF 文件优先文本导出；不可读时要求转换

## 四、升级计划时间桶

为每个证据缺口设计具体可执行的补强任务：

### 半天任务
- 清理项目 README
- 写 ownership 边界说明
- 收集截图/日志
- 列出 10 个 bad case
- 画 pipeline 流程图

### 1 天任务
- 构建小型 eval set
- 添加结构化日志
- 添加可复现 demo 的脚本
- 对比一个 baseline
- 写一份简短技术报告

### 3 天任务
- 添加指标
- 添加 rerank/eval/guardrail/retry 模块
- 添加测试
- 运行消融或对比实验
- 产出 before/after 报告

### 1 周任务
- 构建完整可复现 repo
- 添加 dashboard 或 demo
- 添加文档和示例
- 添加多个 baseline
- 准备最终面试包

### 每个任务的输出格式

```
任务: [具体做什么]
为什么重要: [与 JD 的关系]
时间: [半天/1天/3天/1周]
产出证据: [可交付物]
解锁的简历声明: [补完后可以写的 bullet]
帮助回答的面试题: [对应的高风险面试问题]
```

## 五、项目寻找（弥补技能差距）

当用户缺少 JD 要求的某项技能时，通过开源项目学习来弥补。

### 搜索策略

关键词组合：`[角色关键词] + [机制关键词] + "github" + "from scratch"/"tutorial"/"benchmark"`

### 6 维评分标准

| 维度 | 含义 |
|------|------|
| JD 相关度 | 项目技术栈/场景与目标岗位的匹配程度 |
| 学习价值 | 教机制而非 API 使用（from scratch > wrapper） |
| 可复现性 | 有清晰 README、可独立运行 |
| 证据潜力 | 能产出可验证的代码/报告/截图 |
| 面试价值 | 能回答 JD 对应的高风险面试问题 |
| 改进空间 | 有可修改/扩展的地方（体现主动性） |

### 输出格式

```
候选项目表（按评分排序）：
| 项目 | JD相关度 | 学习价值 | 证据潜力 | 面试价值 | 链接 |
|------|---------|---------|---------|---------|------|

Top 3 推荐：
1. [项目名] — [推荐理由] — [学习计划]
2. [项目名] — [推荐理由] — [学习计划]
3. [项目名] — [推荐理由] — [学习计划]

简历安全声明：[学习后可以合法写入简历的内容]
面试深挖问题：[面试官可能围绕这个项目问什么]
```

**硬边界**：开源学习 ≠ 实习经历。只写用户实际复现、理解、修改、文档化的内容。
