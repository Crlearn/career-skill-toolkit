# JD 分析规则

## 三维度拆解

### 1. 核心能力要求（提取 3 个）

从"岗位要求/任职资格"段落提取，按优先级排序，附 JD 原文依据。

**提取原则**：
- 优先从"必须/要求/必备"类措辞中提取
- 每个能力用一句话概括 + JD 原文引用
- 按出现频率和措辞强度排序

### 2. 隐藏招聘偏好（推断 2 个）

从公司文化描述、团队介绍、加分项中推断未明说的偏好。

**推断来源表**：

| 来源 | 可能推断的偏好 |
|------|--------------|
| "扁平化管理" | 偏好自主驱动、不需要手把手管理的人 |
| "创业氛围" | 偏好能承受不确定性、多面手 |
| "国际化团队" | 偏好有跨文化沟通经验 |
| "快速迭代" | 偏好适应力强、能接受频繁变更 |
| 加分项中的证书/技能 | 虽非必须但有则加分 |

**推断原则**：
- 必须有 JD 中的依据，不凭空猜测
- 标注推断来源（如"来自团队介绍段落"）
- 区分"必须有"和"有则加分"

### 3. 关键词（提取 10 个）

**关键词分类**：
- **硬技能**: 技术工具、编程语言、平台系统（如 Python、AWS、SAP）
- **软技能**: 领导力、沟通、项目管理等（如 stakeholder management）
- **行业术语**: 领域专有名词（如 B2B SaaS、风控模型）
- **资格要求**: 学历、证书、年限、资质（如 PMP、CPA、5年经验）
- **行为动词**: JD 中高频使用的动词（如 drive、lead、optimize）

**关键词优先级**（提取 10 个时按此排序）：
1. 技术栈名词（编程语言、框架、工具名）— 最高权重
2. 方法论/流程名词（敏捷开发、CI/CD、数据驱动）— 高权重
3. 业务领域名词（电商、SaaS、金融）— 中权重
4. 能力素质名词（跨团队协作、项目管理）— 低权重（除非出现 2 次以上）

按重要程度分为：**核心关键词**（必须出现）和 **加分关键词**（出现则加分）。

## 匹配分数计算公式

```
Required Skills 匹配率 = 用户已具备的必需技能数 / 总必需技能数
Preferred Skills 匹配率 = 用户已具备的加分技能数 / 总加分技能数
Overall Match = (Required × 70%) + (Preferred × 30%)
```

**解读标准**：

| 匹配度 | 含义 | 建议 |
|--------|------|------|
| 90-100% | 资历过高 | 可能被视为 flight risk |
| 75-89% | 优秀匹配 | 优先申请 |
| 60-74% | 良好匹配 | 附强力求职信申请 |
| 50-59% | 伸手角色 | 除非特别热爱否则跳过 |
| <50% | 资质不足 | 除非梦想岗位否则跳过 |

**ATS 匹配目标**：优化后简历应达到 **80%+** 匹配度。

## JD 红旗检测

在分析 JD 时同步扫描以下红旗信号：

| 类型 | 红旗措辞 | 隐含含义 |
|------|---------|---------|
| 工作量 | "Wear many hats"、"Fast-paced"、"Hit the ground running" | 可能意味着长工时、角色模糊 |
| 文化 | "Rockstar/Ninja/Guru"、"We work hard, play hard"、"Like a family" | 可能是高压或有毒文化 |
| 薪资 | "Competitive salary"（不说范围）、"Equity-heavy"、"Commission-based" | 现金薪资可能偏低 |

检测到红旗后，在 JD 分析结果中标注，提醒用户注意。

## 关键词匹配度分析

**目标**: 评估简历现有内容与 JD 关键词的匹配情况

**执行方法**:
1. 逐一检查提取的关键词在简历中的出现情况
2. 标记三种状态：
   - ✅ 已覆盖 — 简历中已有明确体现
   - ⚠️ 隐含覆盖 — 经验中有相关内容但未使用关键词表述
   - ❌ 未覆盖 — 简历中完全没有体现
3. 计算整体匹配度百分比
4. 列出需要补充的关键词清单

**输出格式**:
```
匹配度: XX%
| 关键词 | 状态 | 建议 |
|--------|------|------|
| xxx | ✅/⚠️/❌ | ... |
```

## 岗位假设生成（反向匹配）

> 核心理念：不是推荐岗位，而是**生成岗位假设**。基于用户的证据链（见 `references/evidence-chain.md`），反向推导可能匹配的岗位。

### 生成流程

1. 从证据链中提取所有核心能力
2. 将每个能力映射到 2-3 个潜在岗位
3. 用 WebSearch 验证这些岗位的市场需求和典型 JD
4. 为每个候选岗位生成匹配卡片

### 匹配卡片格式

```
┌─────────────────────────────────────┐
│  XX 分  ·  [岗位名称]                │
│                                     │
│  匹配证据：                          │
│  • [证据1] / [证据2] / [证据3]      │
│                                     │
│  主要差距：                          │
│  • [差距1]：[具体描述]               │
│  • [差距2]：[具体描述]               │
│                                     │
│  30天补强计划：                       │
│  • [行动1]：[具体可交付物]           │
│  • [行动2]：[具体可交付物]           │
└─────────────────────────────────────┘
```

### 评分规则

| 分数 | 含义 | 建议 |
|------|------|------|
| 80-100 | 强匹配 | 优先投递，针对性优化简历 |
| 60-79 | 中等匹配 | 可投递，附补强计划 |
| 40-59 | 弱匹配 | 进入观察池，先补强再投 |
| <40 | 不匹配 | 不建议投递，调整方向 |

**低于 60 分不建议投递，只进入观察池。**

### 30天补强计划模板

针对每个差距，设计一个 30 天内可完成的补强行动：

```
| 差距 | 补强行动 | 可交付物 | 预计时间 |
|------|---------|---------|---------|
| 缺少部署经验 | 做 1 个可演示 Demo | GitHub 链接 + 部署截图 | 2 周 |
| 缺少数据分析 | 用真实数据做 1 次分析 | 分析报告 PDF | 1 周 |
| 缺少 XX 证书 | 报名并完成在线课程 | 证书 / 完成证明 | 3 周 |
```

**原则**：每个补强行动必须产出**可交付物**（项目、文档、证书、Demo），不是"学习"或"了解"。

## 多格式 JD 输入支持

分析 JD 时支持以下输入格式：

| 格式 | 处理方式 |
|------|---------|
| 粘贴 JD 文本 | 直接分析 |
| 上传截图 | OCR 识别后分析 |
| 平台导出表（Excel/CSV） | 提取岗位名称和要求列 |
| 浏览器可见页面 | 通过 URL 抓取页面内容 |
| 岗位链接 | WebFetch 获取 JD 页面内容 |

## 求职工作流（五步迭代）

> 每一步都留下文件，可复盘、可更新、可继续迭代。

```
Step 1：简历 / 经历     → 提取证据，构建证据链
Step 2：优势证据链      → 证据→行为→能力→岗位信号
Step 3：岗位假设        → 反向生成匹配卡片，评分排序
Step 4：JD 匹配         → 针对目标 JD 精准优化
Step 5：简历版本 + 面试故事库 → 输出可投递简历 + 面试准备
```

**迭代规则**：
- 每步产出文件保存在用户指定目录
- 用户可随时回到任意步骤修改
- 修改后自动更新下游文件（如证据链更新后，匹配卡片重新评分）
- 所有文件版本可追溯
