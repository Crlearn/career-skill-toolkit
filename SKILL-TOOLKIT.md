本 Skill 覆盖求职流程中的辅助环节，与 `resume-optimizer`、`interview-prep`、`salary-negotiation` 配合使用。

## 一、投递材料包

在填写岗位申请表之前，先生成一份完整的投递材料包供用户审阅：

**材料包结构**：
```
1. 候选人信息：姓名、地点、邮箱、网站
2. 岗位焦点：岗位类型 + 为什么 JD 映射到该类型
3. 定制摘要：2-4 行，连接用户最强证据与 JD
4. 选中证据：3-5 个项目/经历，每个包含匹配原因 + 2-3 条事实 bullet
5. 定制简历要点：用 action + scope + method + outcome，不虚构工具或指标
6. 申请答题草稿：
   - 为什么选择这个岗位
   - 最相关的项目
   - 技术挑战
   - 模糊场景或客户工作流处理
```

**Human Approval Gate（人工审批门控）**：
- 材料包生成后，**必须**请用户审阅
- 用户确认后才能填写表单或提交申请
- 最终提交需要用户对**具体的公司、角色和申请**明确批准
- 如果岗位门户询问法律敏感问题，停止自动填写，请用户直接回答

## 二、申请表填写

### 2.1 核心原则

申请表回答应**直接具体**，不是"塞进文本框的求职信"。每个字段有特定目的，回答该目的即可。

### 2.2 七种问题类型

| 类型 | 示例 | 格式 | 长度 |
|------|------|------|------|
| 经验/背景 | "你有多少年 X 经验？" | [技能] — [X年]。[一句话具体项目锚定] | 2-4 句 |
| Why This Company | "为什么想加入我们？" | [公司特定洞察] + [与真实工作的连接] | <150 词 |
| 作品集 | "附上你的项目" | [项目名]([链接]) — [一行描述+证明点] | 列表格式 |
| 技术技能 | "评估你的 Python 水平" | [技能] — [X年]。[具体用途]。诚实评级 | 2-4 句 |
| 自我介绍 | "介绍一下你自己" | 当前角色 → 相关经历 → 前瞻收尾 | 100-200 词 |
| 情境/行为 | "描述一个你解决的技术难题" | 压缩版 STAR（无标签，自然流畅） | 150-250 词 |
| 观点/愿景 | "你认为未来最重要的技术？" | 真实立场 + 简要理由 | 100-150 词 |

### 2.3 常见错误

| 错误 | 正确 |
|------|------|
| 重复 JD 原文 | 引用公司具体数据/产品决策 |
| 泛泛特质声明（"我学习能力强"） | 描述实际工作，特质自然体现 |
| 过度限定（"虽然我可能没有..."） | 直接陈述，承认差距后转向 |
| 列举无上下文（"React, Vue, Angular..."） | 每个技能附年份+具体用途 |
| 填充无关内容 | 只包含与该岗位相关的内容 |

## 三、投递后跟进

> 国内 BOSS 直聘等平台的专用跟进话术和投递跟踪，见 `career-tracker.md`。本章为通用跟进规则。

### 3.1 跟进节奏规则

| 状态 | 首次跟进 | 后续 | 最大次数 |
|------|---------|------|---------|
| 已投递 | 投递后 7 天 | 每 7 天 | 2 次（之后标记为冷） |
| 已回复 | 1 天内（紧急） | 每 3 天 | 无限制 |
| 面试中 | 面试后 1 天（感谢信） | 每 3 天 | 无限制 |

### 3.2 首次跟进消息框架（3-4 句）

1. **第 1 句**：引用具体角色 + 投递时间
2. **第 2 句**：一个具体的价值点（来自简历或 JD 匹配分析）
3. **第 3 句**：软性询问 + 可用时间（提供具体时间段）
4. **第 4 句（可选）**：简要提及最近的相关项目或成就

**规则**：
- 专业但温暖，**不绝望**
- **禁止**："just checking in"、"just following up"、"touching base"、"circling back"
- 以价值开头，不以请求开头
- 引用该公司的具体内容
- 控制在 150 词以内

**示例**（邮件/Boss直聘消息均可）：
```
您好，

我在 [日期] 投递了 [职位名称] 的申请。想补充一下，
我的 [具体项目/成就] 与该岗位描述的 [具体要求] 高度契合。

很想讨论我的 [X 年经验] 如何为 [公司] 的 [具体团队/项目] 做出贡献。
本周或下周二方便通个电话吗？

谢谢，
[你的姓名]
```

### 3.3 第二次跟进（换角度）

比第一次更短（2-3 句），提供新内容：
- 分享相关洞察、文章或项目更新
- 不重复第一次的内容
- 仍然引用具体角色

### 3.4 冷申请处理（2+ 次跟进无回复）

不再生成新跟进。建议：
- 更新状态为 Discarded（如果岗位似乎已招满）
- 尝试通过其他渠道联系（如 Boss直聘换一个 HR）
- 保留在 Applied 状态但降低优先级

## 四、求职追踪

> 国内平台的投递记录管理和统计面板，见 `career-tracker.md`。本章为通用追踪框架。

### 4.1 申请状态流转

```
Evaluated → Applied → Responded → Interview → Offer
                                         → Rejected
                                         → Discarded
                                         → SKIP
```

| 状态 | 含义 |
|------|------|
| Evaluated | 已评估，待决定是否投递 |
| Applied | 已提交申请 |
| Responded | 公司已回复（尚未面试） |
| Interview | 面试进行中 |
| Offer | 收到 offer |
| Rejected | 被拒绝 |
| Discarded | 主动放弃或岗位关闭 |
| SKIP | 不匹配，不投递 |

### 4.2 追踪表格式

```markdown
| # | 日期 | 公司 | 职位 | 评分 | 状态 | PDF | 报告 | 备注 |
```

### 4.3 模式分析（5+ 申请后）

当有 5 个以上申请进入 Evaluated 以后的状态时，分析：
- **转化漏斗**：各状态的数量和百分比
- **评分 vs 结果**：正面/负面/自筛/待定的平均分
- **原型表现**：哪类岗位转化率最高/最低
- **主要阻碍**：地理限制、技术栈不匹配、资历等
- **推荐最低分**：基于数据驱动的投递门槛

## 五、作品集案例撰写

### 5.1 案例结构（6 部分）

```
1. 概述（项目摘要）
2. 问题（需要解决什么）
3. 过程（如何着手）
4. 解决方案（创建了什么）
5. 结果（影响）
6. 经验教训（会怎么做不同）
```

阅读时间：快速 3-5 分钟，深度 10-15 分钟。

### 5.2 各部分要点

| 部分 | 要点 |
|------|------|
| **概述** | 项目名称、公司、角色、时间线、团队规模、一句话影响 |
| **问题** | 业务背景、用户痛点、目标指标、约束条件 |
| **过程** | 研究、利益相关者、假设、备选方案、决策及理由 |
| **解决方案** | 视觉产出（原型/截图/图表）、关键变更、如何解决问题 |
| **结果** | 量化数据（前后对比表）、业务影响、次要效应 |
| **经验教训** | 什么做得好、会做什么不同、意外挑战、技能发展 |

### 5.3 按角色的侧重点

| 角色 | 侧重 |
|------|------|
| 产品经理 | 策略与优先级、利益相关者管理、指标与结果、技术权衡 |
| UX/设计 | 用户研究、设计过程、视觉产出、可用性改进 |
| 工程师 | 技术架构、问题解决方法、系统设计、代码质量/性能 |
| 营销 | 策略与目标、创意执行、渠道表现、ROI 与归因 |

### 5.4 项目选择原则（技术岗专属）

**核心原则**：不要只做单独 Agent 项目，也不要只做单独 RAG 项目。更好的项目是：
- 同时包含多种技术（如 RAG + Agent）
- 处在同一条业务链路中，形成完整闭环
- 能说清楚业务价值和工程实现
- 具备工程化（API、数据库、缓存、Docker、测试、README）

**项目经历写法模板**（技术岗）：

```
项目名：[项目名称]：基于 [技术A] 与 [技术B] 的 [业务场景] 系统

项目简介：
面向 [场景] 的 [系统类型]，围绕"[业务链路]"构建完整闭环。

技术实现要点：
- 基于 [后端框架] + [数据库] + [缓存] + [消息队列] 搭建后端
- 接入 [LLM API]，实现 [功能1]、[功能2]、[功能3]
- 基于 [Embedding] + [向量库] 实现 [RAG 功能]
- 基于 [Agent 框架] 实现 [Agent 功能]
- 使用 Docker Compose 管理多服务联调与部署

投递建议：
- 简历中直接放 GitHub 链接
- README 要写清楚：项目背景、业务流程、技术架构、启动方式、接口示例、截图或演示
- 如果有在线演示链接，面邀概率会明显提高
```

**面试时必须能讲清**：
- 为什么选择这个业务场景
- 每项技术在项目里解决了什么问题
- 工程化在哪里（框架、数据库、缓存、部署）
- 项目如何从 Demo 变成可投递作品

## 六、跨行求职技能迁移

### 6.1 可迁移技能框架

> 技能迁移的底层方法论见 `references/evidence-chain.md`（证据→行为→能力→岗位信号）。以下为跨行场景的快捷参考：

| 类别 | 技能 |
|------|------|
| **领导与管理** | 团队领导、项目管理、预算管理、战略规划、绩效管理 |
| **沟通** | 演示、书面沟通、客户关系、跨职能协作、谈判、冲突解决 |
| **分析** | 数据分析、问题解决、研究、流程改进、决策、战略思维 |
| **技术/运营** | 流程设计、系统实施、质量保证、培训、供应商管理 |

### 6.2 行业术语翻译示例

**教师 → 企业培训/L&D**：

| 教师术语 | 企业术语 |
|---------|---------|
| 教案 | 培训课程 |
| 学生 | 学员 |
| 课堂管理 | 团队引导 |
| 家长会 | 利益相关者沟通 |
| 考试 | 学习评估 |

**军人 → 企业**：

| 军事术语 | 企业术语 |
|---------|---------|
| 排/单位 | 团队/部门 |
| 任务 | 项目/计划 |
| 情报 | 数据/分析 |
| 简报 | 演示 |
| 部署 | 实施 |

### 6.3 跨行简历策略

1. **以可迁移技能开头的 Summary**："[目标角色] 专业人士，拥有 [X] 年 [可迁移技能] 经验"
2. **使用功能型/混合简历格式**：按技能分组经历，而非严格按时间
3. **构建桥梁经验**：目标领域的志愿工作、自由项目、证书、副业项目

### 6.4 回答"为什么转行？"

**好理由**：兴趣自然演变、通过接触发现热情、技能自然迁移、行业变化、寻求当前路径无法提供的成长

**坏理由（别说）**："我厌倦了"、"我讨厌现在的工作"、"我想赚更多钱"、"看起来更容易"

**故事框架**：
```
[发现]："通过 [经历]，我发现了对 [新领域] 的热情..."
[连接]："我意识到 [旧领域] 的经验给了我 [新领域] 的独特视角..."
[行动]："为了准备转型，我已经 [证书/项目/志愿]..."
[愿景]："我期待将 [可迁移技能] 带到 [目标公司/行业]..."
```

## 七、面试复盘

> 面试结束后，无论成败，结构化复盘是从每次面试中获取最大价值的关键。

### 7.1 复盘模板

每次面试后，在 2 小时内填写以下复盘记录：

```markdown
## 面试复盘 — [公司名] [岗位名]

**日期**：YYYY-MM-DD
**面试轮次**：一面/二面/HR面
**面试官角色**：技术/业务/HR
**结果**：通过/待定/未通过

### 面试官问了什么

| # | 问题 | 类型 | 我的回答 | 回答评分 |
|---|------|------|---------|---------|
| 1 | [原话记录] | 技术/行为/场景 | [实际回答] | ✅/⚠️/❌ |
| 2 | ... | ... | ... | ... |

### 回答评价

**答得好的**：
1. [问题] — [为什么好] — [可以复用到其他面试的点]

**答得不好的**：
1. [问题] — [哪里不好] — [正确回答应该是什么]

**完全没答上来的**：
1. [问题] — [需要补强的知识点] — [补强方式（半天/1天/3天/1周）]

### 补强计划

| 优先级 | 知识点 | 补强方式 | 预计时间 | 解锁的面试题 |
|--------|--------|---------|---------|-------------|
| 🔴 高 | [知识点] | [视频/文档/实操] | [时间] | [对应的面试题] |
| 🟡 中 | ... | ... | ... | ... |

### 下次改进

1. [具体的改进点，如"回答项目经历时先说结论再说细节"]
2. [...]

### 面试官反馈/暗示

- [记录面试官的任何反馈、暗示或特别关注的点]
```

### 7.2 问题类型分类

| 类型 | 特征 | 复盘重点 |
|------|------|---------|
| **技术深度** | "说说 Spring Boot 自动装配原理" | 是否讲清原理+应用场景 |
| **行为面试** | "描述一次你解决冲突的经历" | 是否有完整STAR故事 |
| **场景设计** | "系统突然变慢怎么排查" | 是否有系统化排查思路 |
| **项目追问** | "这个项目你具体做了什么" | 是否与简历声明一致 |
| **反问环节** | "你有什么想问我们的" | 是否问了有深度的问题 |

### 7.3 复盘节奏

| 时间 | 动作 |
|------|------|
| 面试后 2h 内 | 填写复盘记录（记忆最清晰） |
| 面试后 1 天 | 整理补强计划，开始学习 |
| 面试后 3 天 | 礼貌询问面试结果 |
| 下次面试前 | 复习本次复盘中的薄弱点 |

### 7.4 复盘数据积累

当复盘记录达到 3+ 次时，分析模式：
- **高频薄弱点**：哪些知识点反复被问但答不好？
- **回答模式**：是否总是犯同一种错误（如过度展开、答非所问）？
- **面试官关注点**：哪些项目细节最常被追问？
- **改进趋势**：回答评分是否在提升？

### 7.5 与其他 Skill 的协作

| 场景 | 协作 |
|------|------|
| 复盘发现简历声明被深挖暴露 | → SKILL.md 真相边界降级表 |
| 复盘发现技术知识不足 | → SKILL-INTERVIEW.md 技能盘点 |
| 复盘发现需要补强 | → SKILL.md 补强计划时间桶 |
| 复盘发现答案不够亮眼 | → SKILL.md 答案卡片 |
| 复盘后需要更新简历 | → SKILL.md 重新优化简历 |
