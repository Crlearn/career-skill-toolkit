# 面试准备助手

## 触发条件

当用户提出以下需求时，调用此 Skill：
- 面试准备 / STAR 故事库 / 模拟面试
- 预测面试问题 / 准备回答
- 特定公司/岗位的面试策略

## 输入要求

执行前确认：
1. **目标公司和职位**（必须）
2. **JD 文本**（如有，用于精准预测问题）
3. **用户简历**（用于构建 STAR 故事）
4. **优势证据链**（如有，见 `references/evidence-chain.md`，面试故事应基于证据链构建）

## 一、面试准备框架

### Phase 0 — 技能盘点与差距分析（前置步骤）

在正式准备面试前，先帮用户做一次完整的技能盘点，识别差距并提供学习资源。

**Step 0.1：岗位技能要求分析**

基于 JD，梳理该岗位面试时必须掌握的核心技能：

```
### [岗位名] 面试技能要求

| 类别 | 核心技能 | 面试考察深度 | 掌握程度 |
|------|---------|------------|---------|
| 编程语言 | Java | 底层原理 + 高级特性 | ？ |
| 框架 | Spring Boot | 原理 + 源码 | ？ |
| 数据库 | MySQL | 索引 + 事务 + 优化 | ？ |
```

**Step 0.2：用户技能掌握度询问**

> 针对以上核心技能，请你评估自己的掌握程度：
>
> - **精通**：能手写源码、回答底层原理、能在面试中主动展开
> - **熟悉**：用过、能回答常见问题、但深度有限
> - **了解**：知道概念、没深入用过
> - **不会**：完全没接触过
>
> 请逐一告诉我你的掌握程度，不需要谦虚，根据实际情况填写即可。

**Step 0.3：差距分析与学习资源推荐**

根据用户反馈，生成技能差距报告：

```
### 技能差距报告

| 技能 | JD要求 | 你目前的掌握程度 | 差距 | 推荐学习资源 | 预计掌握时间 |
|------|--------|----------------|------|------------|------------|
| Redis | 精通缓存/分布式锁 | 了解基本概念 | 中 | [黑马Redis](https://) + [Redis设计与实现](https://) | 1-2周 |
```

**学习资源推荐规则**：
- 每个差距技能给出 **2-3 个学习资源**，包括：
  - **免费视频课程**（B站/慕课网）
  - **官方文档**（最权威）
  - **书籍**（系统性最强）
  - **项目实战**（直接可写在简历上）
- 学习资源必须**具体**，不能写"学习 Redis"这种模糊建议，要写具体的课程名、书籍名、项目名
- 优先推荐中文资源（因为主要面向国内求职者）
- 如果知识库（`knowledge/` 目录）中有相关资料，优先从知识库推荐

**Step 0.4：面试考察预测**

基于 JD 和技能差距，预测面试中会问的问题：

> 根据 JD 要求和你的技能差距，以下问题**很可能在面试中被问到**，建议你优先准备：
>
> 1. **[某项差距技能相关]**：面试官可能会问"[具体问题]"，你需要能回答"[核心知识点]"
> 2. **[另一项差距技能]**：...
>
> 以下问题你可以自信回答，因为你在简历中已经展示了相关经验：
>
> 1. **[有经验的技能]**：...

**Step 0.5：输出完整报告**

整合 Step 0.1-0.4，输出完整的面试前置准备报告：

```
### [岗位名] 面试准备前置报告

#### 1. 核心技能要求（共 X 项）
[技能要求表]

#### 2. 你的技能掌握度
[掌握度评估]

#### 3. 技能差距分析
[差距表 + 学习资源]

#### 4. 面试高频问题预测
[差距项可能问的问题]
[有经验项可能问的问题]

#### 5. 学习优先级
[按面试频率和差距大小排序的学习计划]
```

**此步骤为可选**，如果用户只想直接准备 STAR 故事，可以跳过 Step 0，直接进入 Phase 1。

## 二、STAR 故事模板

```
SITUATION: "At [Company], we faced [specific challenge]..."
TASK: "I was responsible for [specific ownership]..."
ACTION: "I [specific action 1], [specific action 2], and [specific action 3]..."
RESULT: "As a result, [quantified outcome]. This led to [business impact]."
```

**时长**：90 秒到 2 分钟

### 每个故事准备三个版本
- **完整版**：2 分钟（"Tell me about a time..."）
- **简短版**：60 秒（追问时用）
- **一句话版**：15 秒（"Give me an example"）

### STAR+R 进阶版（资深岗位）

在标准 STAR 基础上增加 **Reflection（反思）**：
- 从这段经历中学到了什么？如果重来会怎么做？
- Reflection 传递资历信号：初级候选人描述发生了什么，资深候选人提炼经验教训

## 三、核心能力故事库（5 类，每类至少 1 个）

| 类别 | 需要的故事 |
|------|-----------|
| **领导力** | 带领团队克服困难、处理冲突、做出不受欢迎的决定、有效授权、培养下属 |
| **问题解决** | 解决复杂技术问题、修复流程缺陷、应对意外障碍、信息不完整时做决策、主动改进 |
| **协作** | 与难相处同事合作、跨职能对齐、建立共识、无职权影响、跨部门协作 |
| **成就** | 超越目标/期望、高压交付、主动出击、最自豪的职业成就 |
| **失败/成长** | 犯错并学习、接受批评、失败后恢复、基于学习改变方法 |

### 故事库构建流程

1. 从简历每条 bullet 出发，扩展为完整 STAR 故事
2. 每个故事映射到 1-3 个能力类别
3. 确保覆盖所有 5 个类别
4. 为每个故事标注：完整版 / 简短版 / 一句话版

## 四、常见问题应对框架

### "Tell me about yourself"（2 分钟电梯演讲）

结构：当前角色 → 核心成就 → 为什么这个岗位 → 为什么现在

### "你最大的弱点是什么？"

**公式**：真实弱点 + 自我认知 + 改进步骤
```
"我有时过于关注细节，可能拖慢进度。我意识到这个问题后，现在会为任务设定时间限制，
并在'够好'和'完美'之间主动寻求反馈。在当前角色中，我也学会了在适当的时候委派工作。"
```

### "你为什么离开现在的公司？"

**保持**：积极、前瞻、简短。不说抱怨，说成长机会。
```
"我在 [公司] 学到了很多，但在寻找 [具体机会]，这在当前路径中看不到。
这个角色恰好提供 [具体内容]。"
```

### "Tell me about a time you failed"

**必须包含**：真实失败（不是谦虚式自夸）+ 学到了什么 + 如何应用所学
```
"在我的第一年作为 PM，我在没有充分用户研究的情况下上线了一个功能。
功能技术上没问题，但只有 10% 的用户采用了它。我深刻认识到，
构建正确的东西比正确地构建东西更重要。从那以后，我从未跳过用户研究——
每次重大功能决策前至少做 10 次用户访谈。"
```

### 薪资问题（面试中被问时）

> 更完整的薪资谈判策略见 `SKILL-SALARY.md`

**先反问**：
```
"我更关注找到合适的角色。能否分享一下这个岗位的预算范围？"
```

**如果被追问**：
```
"基于我的调研和经验，我期望的范围是 $X-$Y，但我愿意讨论整体薪酬包。"
```

## 五、按角色的典型问题

### 产品管理
- "如何确定功能优先级？"
- "如何衡量产品成功？"
- "描述一个从 0 到 1 的产品"

### 工程
- "描述你在 [具体技术] 方面的经验"
- "如何处理技术债 vs 新功能？"
- "描述你解决过的技术挑战"

### 营销
- "如何衡量活动效果？"
- "描述一个效果不佳的活动"
- "如何分配预算？"

### 销售
- "描述你的销售流程"
- "描述一个丢失的 deal 和原因"
- "如何处理异议？"

### AI 应用开发（技术岗专属）

**面试考察四象限**：

| 象限 | 考察内容 | 占比 |
|------|---------|------|
| Python 八股 | 基础语法、异步、装饰器、异常、工程习惯 | 少量 |
| AI 应用基础 | LLM API、Prompt、RAG、Agent、Embedding、Function Calling | 中等 |
| AI 工程化 | FastAPI、数据库、Redis、Celery、Docker、日志、部署 | 中等 |
| AI 项目 | 业务流程、技术选型、难点、优化、可展示程度 | 高 |

**判断**：
- 大厂可能考算法
- 中小厂通常更看技术栈和项目
- 整体难度相当于 Python 后端面试基础上增加 AI 应用，**项目表达很关键**

**常见问题**：
- "你的 RAG 系统 chunk 策略是什么？embedding 模型用的什么？top_k 多少？"
- "Agent 的 tool schema 怎么设计的？调用失败怎么处理？"
- "你的项目怎么从 Demo 变成可部署的？工程化做了哪些？"
- "为什么选这个业务场景？RAG/Agent 分别解决了什么问题？"
- "你的 FastAPI 项目结构是怎样的？怎么处理异步任务？"

**岗位画像**：AI 应用开发实习通常是"全栈 + AI"，工作方式偏 Vibe Coding。Demo 级项目不够，项目必须具备工程化。

## 六、反问面试官的问题

### 问招聘经理
- "这个角色在 30/60/90 天的成功是什么样的？"
- "团队面临的最大挑战是什么？"
- "绩效如何衡量？"
- "团队结构是怎样的？"

### 问团队成员
- "典型的一天/一周是怎样的？"
- "在这里工作最享受什么？"
- "团队如何协作？"
- "新入职的人最需要知道什么？"

### 问高管
- "公司未来一年的战略是什么？"
- "这个团队如何贡献公司目标？"

### 避免问
- ❌ 仅通过 Google 就能查到的信息
- ❌ 薪资/福利（留给 HR）
- ❌ 对公司的负面问题
- ❌ 是非题（问开放式问题）

## 七、面试受众分类（career-ops 方法）

| 受众 | 典型环节 | 评估重点 | 准备策略 |
|------|---------|---------|---------|
| **招聘/HR** | 首轮电话 15-30 分钟 | 动机、薪资、地点/签证、时间线 | 准备 60-90 秒简历概述、薪资范围、时间线 |
| **招聘经理** | 经理面 30-45 分钟 | 为什么这个角色、范围对齐、领导力 | 连接你的叙事 + 团队的具体挑战 |
| **技术/同行** | 技术/系统设计/现场编程 | 技术深度 + 协作能力 | 准备技术 proof points + 反问技术细节 |
| **混合面试组** | 现场/多轮连续 | 跨以上三者的综合 | 准备所有三套，每套精简到 3-5 项 |

**关键原则**：同一证明点在不同面试官面前要**变换角度**，不要逐字重复。不同面试官之间不要自相矛盾。

## 八、面试前公司调研（6 轴框架）

### 必查项目

| 轴 | 调研内容 | 方法 |
|----|---------|------|
| **1. 技术/产品战略** | 产品线、技术栈、工程博客、论文、开源 | 官网、技术博客、GitHub、论文 |
| **2. 近期动态** | 6 个月内：招聘、收购、产品发布、融资 | 新闻、LinkedIn、公告 |
| **3. 工程文化** | 部署节奏、代码仓库、技术栈、远程政策 | GitHub、StackOverflow、员工评价 |
| **4. 潜在挑战** | 扩展问题、可靠性、迁移痛点、竞争 | 分析产品、用户评价、行业报告 |
| **5. 竞争对手** | 护城河、定位、差异化 | 行业分析、竞品对比 |
| **6. 你的角度** | 基于个人画像的独特价值、最相关项目、面试叙事 | 结合简历 + JD + 调研结果 |

### 调研产出

- 面试中可引用的具体事实（"我看到你们最近发布了..."）
- 潜在顾虑和应对准备（"我注意到 X 挑战，我的经验是..."）
- 反问面试官的深度问题（基于调研发现）

## 十一、面试压力测试（衔接 SKILL.md）

完成以上正向准备后，强烈建议进行反向压力测试，暴露简历中的薄弱点。

**推荐流程**：本 Skill（STAR 故事+模拟准备）→ `SKILL.md` 面试拷问5轮 → 答案卡片 → 补强计划

| 压力测试模块 | 位置 | 作用 |
|------------|------|------|
| 面试拷问 5 轮 | `SKILL.md` Phase 1 | 递进式拷问：真相边界→技术深度→JD定向→场景题→风险汇总 |
| 答案卡片 | `SKILL.md` Phase 2 | 高风险问题的危险/及格/强三级回答 |
| 补强计划时间桶 | `SKILL.md` Phase 3 | 按半天/1天/3天/1周分桶输出补强任务 |

**触发词**：对 AI 说"面试拷问""生成答案卡片""补强计划"即可触发。
